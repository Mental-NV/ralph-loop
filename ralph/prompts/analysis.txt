# Analysis Criteria

Evaluate the backlog on these 9 dimensions (0-100 scores):

## 1. Clarity (0-100, weight: 0.15)
Are titles, why statements, deliverables, and exit criteria clear and specific?
- Check for vague language ("implement stuff", "make it work")
- Verify why statements explain rationale
- Ensure deliverables are concrete and actionable
- Confirm exit criteria are specific and measurable

## 2. Completeness (0-100, weight: 0.10)
Do items have all required fields? Are deliverables/exit criteria comprehensive?
- All required fields present (id, title, status, priority, order)
- Optional but important fields populated (why, deliverables, exitCriteria)
- Adequate number of deliverables per item (suggest 2-5)
- Adequate number of exit criteria per item (suggest 2-4)

## 3. Automation-Readiness (0-100, weight: 0.20)
Are validation commands present and executable?
- Validation commands present for most items
- Commands are executable (not pseudo-code)
- Deliverables reference specific files/functions
- Exit criteria can be verified programmatically

## 4. Dependency Structure (0-100, weight: 0.10)
Are dependencies well-defined? Reasonable parallelization?
- Dependencies are necessary and sufficient
- Reasonable parallelization opportunities
- P0 items have minimal dependencies
- No circular dependencies (already validated by schema)

## 5. Risk Awareness (0-100, weight: 0.10)
Are risks identified and mitigation strategies present?
- Risks identified for complex items
- Mitigation strategies included
- Realistic assessment of complexity

## 6. Granularity (0-100, weight: 0.10)
Are items appropriately sized?
- Items completable in one focused session
- Not too many tiny items (suggest 5-15 total)
- Not too few massive items
- Balanced distribution of work

## 7. Priority Alignment (0-100, weight: 0.05)
Do priorities make sense?
- P0 items are infrastructure/foundation
- P1 items are core features
- P2 items are enhancements
- P3 items are polish/nice-to-have
- Dependencies respect priority ordering

## 8. Environment Compatibility (0-100, weight: 0.15)
**IMPORTANT: Dynamically assess environment based on backlog requirements**

Steps to evaluate:
1. Parse backlog items to identify required tech stack (languages, tools, databases, services)
2. Use bash commands to check for required tools in the environment
3. Compare requirements vs. availability
4. Score based on how many requirements are satisfied

Example checks (adapt based on actual backlog):
- If backlog mentions ".NET" or "dotnet" → run `dotnet --version`
- If backlog mentions "Node.js" or "npm" → run `node --version` and `npm --version`
- If backlog mentions "Python" or "pip" → run `python --version` or `python3 --version`
- If backlog mentions "PostgreSQL" → check for `psql` command
- If backlog mentions "Docker" → check for `docker` command
- If backlog mentions "xUnit" → check `dotnet test --help`
- If backlog mentions "Vitest" → check for package.json or `npm list vitest`

Report findings:
- List detected tools with versions
- List missing tools with item IDs that require them
- Provide available_tools and missing_tools arrays

## 9. Agent Capability Alignment (0-100, weight: 0.05)
**IMPORTANT: Dynamically assess required operations**

Steps to evaluate:
1. Parse backlog items to identify required operations
2. Assess if operations are within typical AI agent capabilities
3. Flag operations requiring human approval or elevated permissions
4. Return boolean flags for automated decision-making

Permission flags to set:
- requires_network: true if items need external API calls, package downloads (npm install, dotnet restore)
- requires_destructive_ops: true if items include file deletion, database drops, rm -rf
- requires_system_admin: true if items need sudo, elevated permissions
- requires_external_services: true if items depend on third-party APIs or services
