# Analysis Criteria

Evaluate on these 11 dimensions (0-100 scores):

## 1. Clarity (0-100, weight: 0.125)
Are titles, why statements, deliverables, and exit criteria clear and understandable?
- Check for vague language ("implement stuff", "make it work")
- Verify why statements explain rationale
- Ensure deliverables describe capabilities clearly
- Confirm exit criteria describe outcomes clearly

## 2. Completeness (0-100, weight: 0.083)
Do items have all required fields? Are deliverables/exit criteria comprehensive?
- All required fields present (id, title, status, priority, order)
- Optional but important fields populated (why, deliverables, exitCriteria)
- Adequate number of deliverables per item (suggest 2-5)
- Adequate number of exit criteria per item (suggest 2-4)

## 3. Abstraction Level (0-100, weight: 0.167)
**CRITICAL: Are backlog items high-level and feature-oriented?**

Check for proper abstraction:
- ✅ Deliverables describe WHAT capabilities/features need to exist
- ✅ Exit criteria describe outcomes and behaviors
- ✅ Validation approaches are conceptual (not specific commands)
- ❌ NO specific file names (e.g., "UserController.cs", "src/parser.py")
- ❌ NO specific class/function names (e.g., "Login() method", "parse() function")
- ❌ NO framework-specific details (e.g., "POST /api/auth/login endpoint")
- ❌ NO specific commands (e.g., "Run: curl http://localhost:5000")

Examples of proper abstraction:
- ✅ GOOD: "User authentication capability"
- ❌ BAD: "Create UserController.cs with Login() method"
- ✅ GOOD: "Data persistence layer"
- ❌ BAD: "SQLite database with users table"
- ✅ GOOD: "Verify user authentication flow end-to-end"
- ❌ BAD: "Run: curl -X POST http://localhost:5000/api/auth/login"

Score heavily based on abstraction level:
- 90-100: All items are properly high-level
- 70-89: Most items high-level, some slip into implementation details
- 50-69: Mixed abstraction levels
- 30-49: Most items have implementation details
- 0-29: Items are implementation-specific

## 4. Dependency Structure (0-100, weight: 0.083)
Are dependencies well-defined? Reasonable parallelization?
- Dependencies are necessary and sufficient
- Reasonable parallelization opportunities
- P0 items have minimal dependencies
- No circular dependencies (already validated by schema)

## 5. Risk Awareness (0-100, weight: 0.083)
Are risks identified and mitigation strategies present?
- Risks identified for complex items
- Mitigation strategies included
- Realistic assessment of complexity

## 6. Granularity (0-100, weight: 0.083)
Are items appropriately sized?
- Items completable in one focused session
- Not too many tiny items (suggest 5-15 total)
- Not too few massive items
- Balanced distribution of work

## 7. Priority Alignment (0-100, weight: 0.042)
Do priorities make sense?
- P0 items are infrastructure/foundation
- P1 items are core features
- P2 items are enhancements
- P3 items are polish/nice-to-have
- Dependencies respect priority ordering

## 8. Environment Compatibility (0-100, weight: 0.125)
**IMPORTANT: Read tech stack from docs/ARCHITECTURE.md, then check environment**

Steps to evaluate:
1. Read docs/ARCHITECTURE.md to identify required tools
2. Parse technology stack section for languages, frameworks, databases, testing tools
3. Use bash commands to check if required tools are installed
4. Flag tools that require sudo/admin privileges (Playwright, PostgreSQL, Docker, system packages)
5. Score based on availability of required tools

Example checks (adapt based on ARCHITECTURE.md):
- If architecture specifies ".NET 10" → run `dotnet --version`
- If architecture specifies "Node.js" → run `node --version` and `npm --version`
- If architecture specifies "Python 3.11" → run `python3 --version`
- If architecture specifies "PostgreSQL" → check for `psql` command (requires sudo to install)
- If architecture specifies "Playwright" → check `npx playwright --version` (requires sudo for browser install)
- If architecture specifies "Docker" → check `docker --version` (requires sudo)
- If architecture specifies "xUnit" → check `dotnet test --help`

Report findings:
- List detected tools with versions
- List missing tools (especially those requiring sudo/admin)
- Provide available_tools and missing_tools arrays
- Flag if missing tools block implementation

## 9. Architecture Quality (0-100, weight: 0.083)
**IMPORTANT: Evaluate docs/ARCHITECTURE.md comprehensiveness**

Check if ARCHITECTURE.md exists and contains:
- ✅ Technology stack clearly specified with versions
- ✅ Project structure defined (folder organization, naming conventions)
- ✅ Architectural patterns explained (MVC, Clean Architecture, etc.)
- ✅ Coding standards and conventions documented
- ✅ Testing strategy defined (unit, integration, e2e)
- ✅ Development guidelines (how to add features, structure components)
- ✅ Best practices for the specific tech stack

Score based on completeness:
- 90-100: Comprehensive, actionable architecture document
- 70-89: Good coverage, minor gaps
- 50-69: Basic architecture, missing important sections
- 30-49: Minimal architecture, lacks guidance
- 0-29: Missing or severely incomplete

If ARCHITECTURE.md doesn't exist, score 0 and flag as critical issue.

## 10. Agent Capability Alignment (0-100, weight: 0.042)
**IMPORTANT: Assess required operations from both backlog and ARCHITECTURE.md**

Steps to evaluate:
1. Parse backlog items and ARCHITECTURE.md to identify required operations
2. Assess if operations are within typical AI agent capabilities
3. Flag operations requiring human approval or elevated permissions

Permission flags to set:
- requires_network: true if items need external API calls, package downloads
- requires_destructive_ops: true if items include file deletion, database drops
- requires_system_admin: true if items need sudo, elevated permissions
- requires_external_services: true if items depend on third-party APIs

## 11. Backlog-Architecture Alignment (0-100, weight: 0.083)
**IMPORTANT: Check consistency between backlog and architecture**

Evaluate alignment:
- ✅ Backlog items are feasible given the tech stack in ARCHITECTURE.md
- ✅ Architecture provides enough guidance to implement backlog items
- ✅ No contradictions between backlog and architecture
- ✅ Architecture addresses technical concerns implied by backlog
- ✅ Tech stack in architecture supports all backlog requirements

Examples of misalignment:
- ❌ Backlog mentions "real-time features" but architecture has no WebSocket/SignalR
- ❌ Backlog requires "data persistence" but architecture doesn't specify database
- ❌ Backlog has "frontend" items but architecture only defines backend
- ❌ Architecture specifies Python but backlog implies .NET features
